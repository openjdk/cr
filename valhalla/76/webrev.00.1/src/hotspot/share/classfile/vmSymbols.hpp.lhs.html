 678 // Here are all the intrinsics known to the runtime and the CI.
 679 // Each intrinsic consists of a public enum name (like _hashCode),
 680 // followed by a specification of its klass, name, and signature:
 681 //    template(<id>,  <klass>,  <name>, <sig>, <FCODE>)
 682 //
 683 // If you add an intrinsic here, you must also define its name
 684 // and signature as members of the VM symbols.  The VM symbols for
 685 // the intrinsic name and signature may be defined above.
 686 //
 687 // Because the VM_SYMBOLS_DO macro makes reference to VM_INTRINSICS_DO,
 688 // you can also define an intrinsic's name and/or signature locally to the
 689 // intrinsic, if this makes sense.  (It often does make sense.)
 690 //
 691 // For example:
 692 //    do_intrinsic(_foo,  java_lang_Object,  foo_name, foo_signature, F_xx)
 693 //     do_name(     foo_name, "foo")
 694 //     do_signature(foo_signature, "()F")
 695 // klass      = vmSymbols::java_lang_Object()
 696 // name       = vmSymbols::foo_name()
 697 // signature  = vmSymbols::foo_signature()
 698 //
 699 // The name and/or signature might be a "well known" symbol
 700 // like "equal" or "()I", in which case there will be no local
 701 // re-definition of the symbol.
 702 //
 703 // The do_class, do_name, and do_signature calls are all used for the
 704 // same purpose:  Define yet another VM symbol.  They could all be merged
 705 // into a common 'do_symbol' call, but it seems useful to record our
 706 // intentions here about kinds of symbols (class vs. name vs. signature).
 707 //
 708 // The F_xx is one of the Flags enum; see below.
 709 //
 710 // for Emacs: (let ((c-backslash-column 120) (c-backslash-max-column 120)) (c-backslash-region (point) (point-max) nil t))
 711 //
 712 //
 713 // There are two types of intrinsic methods: (1) Library intrinsics and (2) bytecode intrinsics.
 714 //
 715 // (1) A library intrinsic method may be replaced with hand-crafted assembly code,
 716 // with hand-crafted compiler IR, or with a combination of the two. The semantics
 717 // of the replacement code may differ from the semantics of the replaced code.
 718 //
 719 // (2) Bytecode intrinsic methods are not replaced by special code, but they are
 720 // treated in some other special way by the compiler. For example, the compiler
 721 // may delay inlining for some String-related intrinsic methods (e.g., some methods
 722 // defined in the StringBuilder and StringBuffer classes, see
 723 // Compile::should_delay_string_inlining() for more details).
 724 //
 725 // Due to the difference between the semantics of an intrinsic method as defined
 726 // in the (Java) source code and the semantics of the method as defined
 727 // by the code in the VM, intrinsic methods must be explicitly marked.
 728 //
 729 // Intrinsic methods are marked by the jdk.internal.HotSpotIntrinsicCandidate
 730 // annotation. If CheckIntrinsics is enabled, the VM performs the following
 731 // checks when a class C is loaded: (1) all intrinsics defined by the VM for
 732 // class C are present in the loaded class file and are marked;
 733 // (2) an intrinsic is defined by the VM for all marked methods of class C;
 734 // (3) check for orphan methods in class C (i.e., methods for which the VM
 735 // declares an intrinsic but that are not declared for the loaded class C.
 736 // Check (3) is available only in debug builds.
 737 //
 738 // If a mismatch is detected for a method, the VM behaves differently depending
 739 // on the type of build. A fastdebug build exits and reports an error on a mismatch.
 740 // A product build will not replace an unmarked library intrinsic method with
 741 // hand-crafted code, that is, unmarked library intrinsics are treated as ordinary
 742 // methods in a product build. The special treatment of a bytecode intrinsic method
 743 // persists even if the method not marked.
 744 //
 745 // When adding an intrinsic for a method, please make sure to appropriately
 746 // annotate the method in the source code. The list below contains all
 747 // library intrinsics followed by bytecode intrinsics. Please also make sure to
 748 // add the declaration of the intrinsic to the approriate section of the list.
